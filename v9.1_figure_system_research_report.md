## 一、TCSVT 顶刊图表特征分析（What）

### 1.1 IEEE 官方技术规范

| 参数 | 要求 |
|------|------|
| 单栏宽度 | 3.5 inch (88.9mm) |
| 双栏宽度 | 7.16 inch (182mm) |
| 线稿 DPI | ≥ 600 |
| 彩色图 DPI | ≥ 300 |
| 灰度图 DPI | ≥ 300 |
| 最小字号 | 缩放后 ≥ 6pt |
| 首选格式 | 矢量 (EPS/PDF) 或高分辨率 TIFF |
| 色彩模式 | RGB（在线）/ CMYK（印刷） |

### 1.2 TCSVT 图表的共性特征（从论文反推）

通过对 TCSVT 高引论文的图表分析，总结出以下 **7 大共性特征**：

#### (1) 主图 + 支图分层策略
- **主图 (Fig.1)**：通常是 **Teaser Figure**，一张图概括整个方法的核心思想
  - 放在 Introduction 中，使用 `figure*` 双栏环境
  - 信息密度高但层次清晰，读者 30 秒内能抓住核心贡献
  - 通常包含：输入→核心处理→输出的宏观流程
- **支图 (Fig.2-5)**：展示具体创新点的细节
  - 模块详图：某个关键子模块的内部结构
  - 消融可视化：去掉/加入某个组件的效果对比
  - 定性结果：与 SOTA 方法的视觉对比

#### (2) 信息层次有序组织
- **视觉流方向一致**：从左到右或从上到下，绝不混乱
- **模块级连接**：只在模块之间画箭头，不在细粒度组件之间连线
- **分组明确**：功能相关的元素用虚线框/浅色背景分组
- **标注精简**：每个模块的标签 ≤ 3 个单词，详细说明放图注

#### (3) 配色方案专业克制
- **主色调**：蓝-灰-白为主基调，辅以 1-2 个强调色
- **色盲友好**：避免红-绿对比，使用蓝-橙或蓝-红
- **饱和度低**：填充色使用浅色（opacity 0.2-0.4），边框使用深色
- **一致性**：同一论文所有图表使用统一配色方案

#### (4) 排版精确对齐
- 模块间距均匀（2-5mm）
- 同行模块等高，同列模块等宽
- 箭头粗细统一，箭头样式一致
- 面板标签 (a)(b)(c) 统一放在左上角，粗体 10-12pt

#### (5) 数据图表规范
- 坐标轴标签 6-8pt，轴标题 8-10pt
- 数据线 0.5-1.0pt，坐标轴 0.3-0.5pt
- 每条线有独特标记符号 + 颜色双重编码
- 图例直接标注在数据线旁，而非独立图例框

#### (6) 矢量输出 + 可编辑
- 所有图表输出矢量格式（PDF/EPS/SVG）
- 文字可选中、可搜索
- 缩放不失真

#### (7) 一图一故事
- 每张图只传达一个核心观点
- 如果 30 秒内无法理解，说明需要拆分
- 图注完整自解释，不依赖正文也能理解

## 二、业界图表生成系统架构分析（How）

### 2.1 SciFig（arXiv 2601.04390）— 多Agent + 层次化布局 + CoT反馈

**架构**：4 个专业 Agent 流水线 + 闭环反馈

```
文本 → Description Agent → 层次结构 H
         ↓
       Layout Agent → 初始布局 L₀
         ↓
    ┌──→ 渲染 → Feedback Agent → 反馈 FB_t ──┐
    │         ↓                                │
    └── Layout Agent (CoT推理修正) ←──────────┘
         ↓ (收敛后)
       Component Agent → 最终矢量图表
```

**核心创新**：
1. **层次化布局**：先排模块，再排模块内组件。模块间连线（非组件间连线），减少视觉混乱
2. **CoT 反馈循环**：Feedback Agent 独立审查（避免确认偏差）→ Layout Agent 用思维链推理修正
3. **6维评分体系**：从 2219 张真实科学图表中提取评价标准

**效果**：
- 质量得分 70.1/100（超越 GPT-5-Image、Gemini-2.5）
- 达到人类专家手工设计 81.3% 的水平
- 生成时间 9.7 分钟（人类 227.5 小时），1407 倍加速

### 2.2 AutoFigure（ICLR 2026）— 双Agent + Review-Refine循环

**架构**：生成 Agent + 评估 Agent 闭环

```
文本/PDF → Extractor → 方法论文本
         ↓
    ┌──→ Generation Agent → SVG/XML 代码
    │         ↓
    │    Evaluation Agent → 评分(0-10) + 具体反馈
    │         ↓
    └── (分数 < 9.0 则继续迭代，最多 5 轮)
         ↓
    最终 SVG / mxGraph XML (draw.io 可编辑)
```

**核心创新**：
1. **质量阈值驱动**：默认 9.0 分才通过，否则迭代
2. **双格式输出**：SVG（代码级可编辑）+ mxGraph XML（draw.io GUI 可编辑）
3. **可选 AI 图像增强**：生成后用 Gemini Flash 做美学后处理

**技术栈**：Python SDK + Flask API + Next.js UI + Playwright 渲染

### 2.3 PaperBanana — 5 Agent 流水线 + Critic 反馈

**架构**：5 个专业 Agent 顺序协作

```
用户输入 → Retriever → 顶会参考图检索
              ↓
           Planner → 结构描述（组件/连接/分组/布局）
              ↓
           Stylist → 配色/排版/线宽/间距（对标目标期刊）
              ↓
           Visualizer → 渲染（数据图→Matplotlib代码，示意图→图像）
              ↓
        ┌──→ Critic → 逐点比对源文本 → 通过/不通过
        │         ↓ (不通过)
        └── Visualizer (针对性修正)
```

**核心创新**：
1. **Retriever 检索参考图**：从 NeurIPS/ICML/CVPR 论文图库中检索最相关示例，做上下文学习
2. **Stylist 对标期刊风格**：根据目标 venue 自动选择配色/排版
3. **数据图输出可执行代码**：Visualizer 生成 Matplotlib 代码而非直接生图，消除数值幻觉
4. **Critic 逐点比对**：将生成结果与源文本逐点对照，确保忠实度

### 2.4 academic-figure-generator — Claude + NanoBanana

**架构**：论文上传 → Claude 分析 → Prompt 生成 → AI 图像生成

**核心创新**：
1. **50+ 预设学术配色方案**（含色盲友好方案）
2. **图生图编辑**：基于已生成图片叠加文字指令微调
3. **配套 Agent Skill**：可在 Cursor/Claude Code 中使用

## 三、当前系统差距分析（Gap）

### 3.1 当前系统架构

```
project_data + experiment_data
         ↓
  figure_generator.py (纯 matplotlib，硬编码)
         ↓
  5 类图表一次性生成（无 plan/evaluate/iterate）
         ↓
  PDF 输出
```

### 3.2 逐维度差距对比

| 维度 | 当前系统 (v9.0) | 顶刊标准 | 差距等级 |
|------|----------------|---------|---------|
| **Plan 阶段** | ❌ 无。硬编码模块坐标和内容 | ✅ 从论文文本自动提取结构，层次化布局 | 🔴 严重 |
| **Generate 阶段** | ⚠️ 纯 matplotlib，配色写死，布局硬编码 | ✅ 多模态渲染（矢量+图像+文本），风格对标目标期刊 | 🟡 中等 |
| **Evaluate 阶段** | ❌ 无。生成即完成 | ✅ 独立视觉审查，多维评分，逐点比对 | 🔴 严重 |
| **Iterate 阶段** | ❌ 无。一次性生成 | ✅ 反馈→修正循环，质量阈值驱动 | 🔴 严重 |
| **主图/支图分层** | ❌ 无。所有图同等对待 | ✅ 主图抓主要矛盾 + 支图展示创新点 | 🔴 严重 |
| **配色方案** | ⚠️ 6 色浅色系，无学术规范对标 | ✅ 低饱和度+色盲友好+venue 自适应 | 🟡 中等 |
| **信息层次** | ❌ 所有模块平铺，无分组/无层次 | ✅ 模块级连接，功能分组，视觉流引导 | 🔴 严重 |
| **排版对齐** | ⚠️ 手动坐标，容易不对齐 | ✅ 网格对齐，间距均匀，标签统一 | 🟡 中等 |
| **输出格式** | ✅ PDF 矢量 | ✅ PDF/EPS/SVG 矢量 | 🟢 基本达标 |
| **尺寸规范** | ⚠️ figsize=(7,3.5) 硬编码 | ✅ 自动适配 venue 单栏/双栏宽度 | 🟡 中等 |
| **TikZ 生成** | ⚠️ 有 LLM 生成 + 4 重后处理，但无迭代评价 | ✅ 需要 evaluate→iterate 闭环 | 🟡 中等 |

### 3.3 核心问题总结

**问题 1：无 Plan → 信息无序**
当前 `figure_generator.py` 把所有模块坐标写死在代码里，不从论文内容中提取结构。导致：
- 无法适配不同论文的方法架构
- 模块之间没有功能分组
- 信息流方向不清晰

**问题 2：无 Evaluate → 质量盲盒**
生成完就结束，不知道图好不好。导致：
- 文字可能重叠/截断
- 箭头可能交叉/混乱
- 配色可能不协调
- 信息密度可能过高或过低

**问题 3：无 Iterate → 无法改进**
即使知道有问题，也没有修正机制。导致：
- 每次只能人工检查、手动修改代码
- 无法自动收敛到满意质量

**问题 4：无主图/支图策略 → 详略失当**
所有图表同等对待，没有：
- 主图概括全局（Teaser Figure）
- 支图展示创新点细节
- 根据论文结构自动决定需要哪些图

## 四、v9.1 图表系统升级方案

### 4.1 目标架构：Plan → Generate → Evaluate → Iterate (PGEI)

```
                    ┌─────────────────────────────────────────┐
                    │           Figure Planner Agent           │
                    │  输入: 论文方法文本 + 创新点 + venue      │
                    │  输出: 图表清单 + 每张图的结构描述         │
                    │        (模块/连接/分组/层次/主图vs支图)    │
                    └──────────────────┬──────────────────────┘
                                       ↓
                    ┌─────────────────────────────────────────┐
                    │          Figure Stylist Agent            │
                    │  输入: venue + 图表结构                  │
                    │  输出: 配色方案/字体/线宽/布局参数         │
                    │        (对标 TCSVT/TIP/CVPR 风格)        │
                    └──────────────────┬──────────────────────┘
                                       ↓
              ┌────────────────────────────────────────────────┐
              │              Generate Phase                     │
              │  ┌─────────────┐  ┌──────────────────────────┐ │
              │  │ Matplotlib  │  │ TikZ Generator           │ │
              │  │ (数据图表)   │  │ (架构/流程示意图)         │ │
              │  │ 柱状图/折线  │  │ 模块图/流程图/Teaser     │ │
              │  └──────┬──────┘  └────────────┬─────────────┘ │
              │         └──────────┬───────────┘               │
              └────────────────────┼───────────────────────────┘
                                   ↓
              ┌────────────────────────────────────────────────┐
              │           Evaluate Phase                        │
              │  ┌──────────────────────────────────────────┐  │
              │  │ Figure Critic Agent (视觉模型)            │  │
              │  │  输入: 生成的图 + 原始结构描述             │  │
              │  │  评分: 6 维度 (0-10)                      │  │
              │  │  ① 信息准确性 ② 布局合理性 ③ 配色协调性   │  │
              │  │  ④ 信息层次 ⑤ 排版对齐 ⑥ 学术规范性       │  │
              │  │  输出: 总分 + 各维度分 + 具体修改建议      │  │
              │  └──────────────────────────────────────────┘  │
              └────────────────────┬───────────────────────────┘
                                   ↓
                          ┌────────────────┐
                          │ 总分 ≥ 8.0 ?   │
                          │  且无单项 < 6  │
                          └───┬────────┬───┘
                           Yes│        │No (最多 3 轮)
                              ↓        ↓
                         最终输出   回到 Generate Phase
                                   (携带 Critic 反馈作为修正指令)
```

### 4.2 核心模块设计

#### Module 1: FigurePlanner — 图表规划器

**职责**：从论文内容中提取图表需求，生成结构化的图表计划

```python
class FigurePlanner:
    def plan(self, paper_content: dict, venue: str) -> FigurePlan:
        """
        输入: 论文内容(方法描述/创新点/实验数据) + 目标venue
        输出: FigurePlan 包含:
          - figure_list: 图表清单
            - fig_id: "fig1_main_architecture"
            - fig_type: "teaser" | "architecture" | "module_detail" | "ablation" | "comparison"
            - fig_role: "main" | "supporting"  # 主图 vs 支图
            - content_spec: 结构化内容描述
              - modules: [{id, label, group, is_innovation}]
              - connections: [{from, to, label}]
              - groups: [{id, label, module_ids}]
              - layout_hint: "left_to_right" | "top_to_bottom" | "grid"
            - size_spec: "single_column" | "double_column"
            - priority: 1-5 (决定生成顺序和资源分配)
        """
```

**关键设计**：
- 主图 (Fig.1) 必须是 Teaser，概括整个方法的核心思想

- 数据图表（消融/对比）从实验数据自动生成

#### Module 3: FigureCritic — 图表评审器

```python
class FigureCritic:
    def evaluate(self, figure_path: str, spec: dict) -> CriticReport:
        """
        使用视觉模型 (GLM-5V-turbo, 国内可用无需梯子) 评审生成的图表
        
        评分维度 (6维, 每维 0-10):
        1. accuracy: 图表内容是否与论文描述一致
        2. layout: 布局是否合理，模块是否对齐，间距是否均匀
        3. color: 配色是否协调，是否色盲友好，是否符合venue风格
        4. hierarchy: 信息层次是否清晰，主次是否分明
        5. typography: 字体大小是否可读，标签是否完整
        6. compliance: 是否符合IEEE规范（尺寸/DPI/格式）
        
        输出: CriticReport
          - scores: {accuracy, layout, color, hierarchy, typography, compliance}
          - overall: 加权总分
          - issues: [{dimension, severity, description, fix_suggestion}]
          - passed: bool (overall >= 8.0 and no dimension < 6)
        """
```

## 五、环境约束与技术选型

### 5.1 环境约束

| 约束 | 说明 |
|------|------|
| **无梯子** | 不能访问 Gemini/DALL-E/OpenAI Image 等海外 API |
| **不用大模型直接生图** | 排除 AI 图像生成（像素级），只用代码渲染（矢量级） |
| **Node.js v18 可用** | 可用于 SVG 渲染、Playwright 截图、PDF 转换等 |
| **国内视觉模型可用** | glm-5v-turbo / glm-4.6v（智谱），已集成 `query_vision()` |
| **国内文本模型可用** | glm-5.1 / qwen3.7-max 等，已集成 `query()` |

### 5.2 修正后的技术路线

**核心原则：LLM 生成的是代码（SVG/TikZ/Matplotlib），不是像素。视觉模型只用于评审，不用于生成。**

```
                    ┌─────────────────────────────────────────┐
                    │           Figure Planner                 │
                    │  LLM: glm-5.1 / qwen3.7-max             │
                    │  输入: 论文方法文本 + 创新点 + venue      │
                    │  输出: JSON 结构化图表计划                │
                    └──────────────────┬──────────────────────┘
                                       ↓
                    ┌─────────────────────────────────────────┐
                    │           Code Generator                 │
                    │  LLM: glm-5.1 / qwen3.7-max             │
                    │  输出: Matplotlib 代码 / TikZ 代码 / SVG │
                    │  （纯代码，不依赖任何图像生成 API）       │
                    └──────────────────┬──────────────────────┘
                                       ↓
                    ┌─────────────────────────────────────────┐
                    │           Code Renderer                  │
                    │  Python: exec(matplotlib_code)           │
                    │  或 Node.js: SVG → PNG/PDF               │
                    │  或 LaTeX: TikZ → PDF                    │
                    │  （纯本地渲染，零网络依赖）               │
                    └──────────────────┬──────────────────────┘
                                       ↓
                    ┌─────────────────────────────────────────┐
                    │           Figure Critic                  │
                    │  视觉模型: glm-5v-turbo (国内, 无需梯子) │
                    │  输入: 渲染后的 PNG + 原始结构描述        │
                    │  输出: 6维评分 + 具体修改建议             │
                    └──────────────────┬──────────────────────┘
                                       ↓
                              总分 ≥ 8.0 → 最终输出
                              总分 < 8.0 → 反馈给 Code Generator 重新生成
```

### 5.3 与海外方案的对比

| 方面 | PaperBanana/AutoFigure | 我们的方案 (v9.1) |
|------|----------------------|-------------------|
| 图表生成 | Gemini Flash 生图 / SVG 代码 | **纯代码渲染**（Matplotlib/TikZ/SVG） |
| 视觉评审 | GPT-4V / Gemini Pro Vision | **glm-5v-turbo**（国内可用） |
| 文本生成 | GPT-4 / Claude | **glm-5.1 / qwen3.7-max** |
| 渲染引擎 | Playwright + Chromium | **Python matplotlib + Node.js SVG + LaTeX TikZ** |
| 网络依赖 | 需要访问海外 API | **零海外 API 依赖** |
| 可编辑性 | SVG / draw.io XML | **PDF 矢量 + Matplotlib 源码 + TikZ 源码** |

### 5.4 渲染引擎选型

| 图表类型 | 渲染引擎 | 理由 |
|---------|---------|------|
| 架构图/流程图 (Teaser) | **TikZ** (LaTeX) | 与论文正文排版完美融合，矢量精确 |
| 模块详图 | **SVG 代码** (Node.js 渲染) | 灵活度高，LLM 生成 SVG 比 TikZ 更可靠 |
| 数据对比图 | **Matplotlib** (Python) | 数据驱动，精确可控，可复现 |
| 消融实验图 | **Matplotlib** (Python) | 同上 |
| 定性结果对比 | **Matplotlib subplot** | 图像拼接+标注 |

### 5.5 Node.js 在渲染链中的角色

```
SVG 代码 (LLM 生成)
    ↓
Node.js + sharp/resvg-js → PNG (300 DPI)
    ↓
glm-5v-turbo 评审 PNG
    ↓
(通过) → SVG → PDF (矢量嵌入 LaTeX)
(不通过) → 反馈给 LLM 修正 SVG 代码
```

Node.js 可用库：
- `@resvg/resvg-js`: 高性能 SVG → PNG 渲染（Rust 核心）
- `sharp`: 图像处理（裁剪、缩放、格式转换）
- `playwright`: 浏览器截图（复杂 HTML/SVG 渲染兜底）

## 七、总结

**核心发现**：当前系统最大的差距不是"画得不好看"，而是**缺少 Plan-Evaluate-Iterate 闭环**。

所有成功的图表生成系统（SciFig/AutoFigure/PaperBanana）都遵循同一个范式：
1. **Plan**：从文本提取结构，层次化布局
2. **Generate**：多模态渲染，风格对标目标期刊
3. **Evaluate**：独立视觉审查，多维评分
4. **Iterate**：反馈驱动修正，质量阈值收敛

v9.1 的核心任务就是为当前系统注入这个 PGEI 闭环，而不是简单地换一个更好看的配色或模板。

**技术路线已适配环境约束**：
- 全程使用国内 API（智谱 glm-5.1 生成代码 + glm-5v-turbo 视觉评审），零梯子依赖
- LLM 只生成代码（Matplotlib/TikZ/SVG），不生成像素图像
- Node.js v18 本地渲染 SVG → PNG，供视觉模型评审
- 最终输出矢量 PDF，嵌入 LaTeX 编译
