## 五、关键发现与洞察

### 5.1 发现 1: 公式转换是主要瓶颈

**现象**: 两个案例中，公式语法错误占比最高（Case 1: 96%, Case 2: 100%）

**根因**: `latex_converter.py` 在 Markdown → LaTeX 转换时：
- 行内公式 `$...$` 边界识别不准确
- 显示公式 `$$...$$` 与 `\[...\]` 混用导致括号不匹配
- 公式编号 `\eqno` 插入位置错误

**影响**: 导致大量 `Missing $ inserted` 和 `Extra }` 错误

**建议**: 
1. 增强 `latex_converter.py` 的公式解析逻辑
2. 添加公式语法预检查（在编译前）
3. 考虑使用 LLM 辅助修复复杂公式错误

### 5.2 发现 2: BibTeX 生成不稳定

**现象**: Case 1 的 `.bib` 文件为空（0 条目），Case 2 正常（47 条目）

**根因**: Phase 7.8（BibTeX 生成）可能因 LLM 调用失败或解析错误导致生成失败

**影响**: 所有 `\cite{}` 命令无法解析，产生大量 `undefined citation` 警告

**建议**:
1. 在 Phase 7.8 后添加 BibTeX 验证
2. 如果 `.bib` 为空，触发重新生成
3. 添加引用条目完整性检查

### 5.3 发现 3: XeLaTeX 容错机制掩盖问题

**现象**: 尽管有 46-49 个错误，XeLaTeX 仍成功生成 PDF（10-14 页）

**原因**: XeLaTeX 的 `-interaction=nonstopmode` 模式会忽略错误继续编译

**影响**: 
- PDF 虽然生成，但内容可能有缺失或排版错误
- 用户可能误以为"编译成功=质量合格"

**建议**:
1. Phase 8.5 验证器应标记"带错编译"为警告
2. 在验证报告中明确标注错误数量
3. 提供错误位置定位（行号）

### 5.4 发现 4: 自动修复能力有限

**现象**: 两个案例的自动修复率均为 0%

**原因**: 
- 当前自动修复仅支持 3 种简单模式（表格溢出、缺失引用、图片缺失）
- 实际问题主要是公式语法错误和 TikZ 配置错误，超出规则修复范围

**建议**:
1. 集成 LLM 辅助修复（使用 api_client）
2. 针对高频问题（公式错误）设计专门修复策略
3. 建立错误-修复知识库
