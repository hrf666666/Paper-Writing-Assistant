## 五、5 个新模块逐一分析

### 5.1 VERIFY 独立验证层 (`agent/verifier.py`)

**功能**: 8 项纯代码检查（引用完整性/数据一致性/公式语法/段落去重/残留标记/章节引用/结构完整性/符号一致性），零 LLM 成本。

| 优点 | 缺点 |
|------|------|
| 零 LLM 成本，每次验证 <1ms | V1 引用检查对空格引用（` `）无法识别为有效引用 |
| 独立于 REFLECT，不受 LLM 自评偏差影响 | V2 数值一致性检查仅做正则匹配，无法理解语义等价 |
| 成功检测出 `<formula>`/`<citation>` 残留 | 对 LaTeX `\cite{}` 格式引用未做支持 |
| Abstract 3 轮验证确保 100 分通过 | 章节级验证无法检测跨章节数值不一致 |

**实际效果**: 成功运行 11 次验证，正确识别了残留标记问题。Conclusion/Abstract 达到 100 分。

### 5.2 有序门控流水线 (`agent/ordered_gate.py`)

**功能**: P0 格式→P1 一致性→P2 质量，高优先级失败时跳过后续，含死循环保护。

| 优点 | 缺点 |
|------|------|
| 优先级分级合理，P0 阻断后续避免浪费 | 当前 P0 阈值过严（任何 `[?]` 都阻断），导致 P1/P2 被跳过 |
| 自动修复后重新验证机制有效 | 自动修复仅清理标记，无法修复结构性问题 |
| 执行时间 <0.1s，极低成本 | 修复后 score=95.0 仍被阻断，说明阈值设置不合理 |
| 死循环保护（max_retries=3）正常工作 | 缺少可配置的阈值参数 |

**实际效果**: 正确识别了 P0 格式问题，自动修复了 4 个问题。但 P1/P2 门控被跳过，未能发挥完整作用。

### 5.3 恒定大小上下文管理 (`agent/bounded_context.py`)

**功能**: 3 层记忆（工作/短期/长期），固定 5000 字符预算，结构化摘要提取。

| 优点 | 缺点 |
|------|------|
| 固定预算避免上下文爆炸 | 实际未在章节生成 prompt 中使用 |
| 章节摘要自动提取和压缩 | 符号提取依赖正则，覆盖率有限 |
| 前序章节摘要传递机制完善 | 与 dispatcher 的集成不够深入 |
| 项目简介和创新点存储结构化 | 恢复时需从 chapters 重建，增加启动时间 |

**实际效果**: 成功存储了 5 个章节摘要和项目简介。但由于章节生成编排器（ch1-ch5）未修改 prompt 模板来利用 BoundedContext 传递的上下文，实际效果有限。

### 5.4 章节级状态机 (`agent/chapter_state_machine.py`)

**功能**: outline→draft→review→revision→final 生命周期管理，含阶段门控和重试限制。

| 优点 | 缺点 |
|------|------|
| 状态转换严格，防止跳跃 | Related Work 触发了重试（score=87.5 < 50 阈值），但实际未执行重试 |
| 每阶段记录 score 和历史 | 重试逻辑与 dispatcher 任务重试冲突 |
| 进度追踪可视化（progress 属性） | 状态机与主循环的集成点不够多 |
| 死循环保护（max_retries=3） | `get_next_phase()` 在某些边界条件下返回 None |

**实际效果**: 成功注册了 7 个章节，记录了各章节的 VERIFY 分数。Related Work 触发了重试机制（"阶段 draft 未通过 (score=87.5 < 50)，重试 1/3"），但由于 dispatcher lambda 参数错误，重试未实际执行。

### 5.5 ToolTrace 反捏造 (`agent/tool_trace.py`)

**功能**: 追踪搜索/引用工具调用结果，验证论文中引用的真实性。

| 优点 | 缺点 |
|------|------|
| 记录所有 Semantic Scholar 搜索结果 | 论文中无 `[N]` 格式引用，导致 0/0 验证 |
| 引用关键词与搜索结果交叉验证 | 仅支持 `[N]` 格式，不支持空格引用 |
| 统计信息完整（verified/unverified/total） | 未在章节生成过程中实时记录工具调用 |
| 可扩展的 verify_claims 接口 | 与 verifier.py 的 V1 检查有功能重叠 |

**实际效果**: Phase 7.18 报告显示 "引用验证: 0/0 已验证"，因为最终输出中没有 `[N]` 格式引用。ToolTrace 的潜力未充分发挥。

## 六、架构升级优缺点

### 6.1 从 3 阶段到 4 阶段循环

```
旧: THINK → EXECUTE → REFLECT        (LLM自评自判，无独立验证)
新: THINK → EXECUTE → VERIFY → REFLECT (纯代码验证独立于LLM评估)
```

| 优点 | 缺点 |
|------|------|
| VERIFY 独立于 REFLECT，避免 LLM "自己判自己作业" | 增加了每章节 ~1ms 验证时间（可忽略） |
| 纯代码检查零 LLM 成本 | VERIFY 结果对 REFLECT 决策的影响有限（仅 score<40 才强制重试） |
| 可复现、可调试、可单元测试 | 4 阶段增加了系统复杂度和调试难度 |
| 为后续更复杂的验证（如编译检查）预留接口 | 当前 VERIFY 检查项覆盖面不够广 |

### 6.2 Phase 7 增强（门控 + ToolTrace + 自动修复）

| 优点 | 缺点 |
|------|------|
| 输出前多道质量关卡 | Phase 7 耗时翻倍（7min→14.3min） |
| 自动修复机制减少了人工干预 | 自动修复能力有限（仅清理标记） |
| 门控报告可追溯 | P0 阈值过严导致 P1/P2 被跳过 |

### 6.3 整体架构评价

**最大优势**: 建立了"不信任 LLM 自述"的验证哲学。VERIFY 层和 ToolTrace 从工具结果而非 LLM 叙述中提取事实，这是从 auto_research_agent 借鉴的核心理念。

**最大劣势**: 新模块与现有编排器（ch1-ch5 skill_orchestrators）的集成深度不够。BoundedContext 和 ToolTrace 的数据未被章节生成 prompt 利用，形成了"有数据但不用"的浪费。
