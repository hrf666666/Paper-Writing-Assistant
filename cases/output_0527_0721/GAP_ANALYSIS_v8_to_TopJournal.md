# v8.0 → 顶刊论文差距分析与改进路线图

> 生成时间: 2026-05-27
> 基于对 v7.0/v8.0 系统的完整代码审查、pipeline 运行数据和质量评估

---

## 核心论点

> **生成论文这个 task 本身不难。顶层设计（3个创新点/主要工作）是关键。**
> 如果顶层设计正确，每个章节的内容设计比顶层更简单。

当前系统最核心的瓶颈不在"生成"，而在**"想清楚要写什么"**。

## 一、差距总览：v8.0 输出 vs 顶刊论文

| 维度 | v8.0 现状 | 顶刊要求 | 差距等级 |
|------|----------|---------|:--------:|
| **创新点提炼** | LLM 从代码中提取，可能遗漏/泛化 | 清晰、具体、可量化的3个贡献 | **Critical** |
| **叙事连贯性** | motivation_thread 存在但不强制执行 | 贯穿全文的故事线，每个段落都在推进论点 | **High** |
| **实验可信度** | 数据从代码/报告中提取，无法验证真实性 | 可复现的实验，标准benchmark，公开数据集 | **Critical** |
| **引用质量** | 95.1% 验证通过，但引用匹配精度不高 | 精确引用最相关的工作，每个引用有明确目的 | **Medium** |
| **数学严谨性** | LaTeX 公式生成但无验证 | 推导完整、符号一致、定理可证 | **High** |
| **写作风格** | 模板化，缺乏"声音" | 有学术声音，行文流畅，逻辑紧凑 | **Medium** |
| **跨章节一致性** | 0 critical，但有数值不一致 | 完美一致，Abstract-正文完全吻合 | **Low-Medium** |
| **图表质量** | 消融可视化存在，但粗糙 | 专业图表，信息密度高，视觉精良 | **Medium** |

## 二、从无到有：必须先解决的问题（P0）

### 2.1 顶层设计：创新点确定机制不够强

**现状**：`project_analyzer.extract_innovation_points()` 让 LLM 从代码中提炼创新点，但：
- 创新点可能过于泛化（如"提出了一种新的损失函数"而不是"提出了一种基于 XX 理论的 YY 损失，在 ZZ 条件下理论收敛"）
- 没有验证创新点的**新颖性**（是否已被别人提出过）
- 没有验证创新点的**可行性**（是否有足够的代码支撑）
- 创新点之间缺乏**递进关系**（好的论文创新点通常是层层递进的）

**改进方案**：

```
Phase 0.1: 代码分析 → 候选创新点池（当前有）
Phase 0.2: [新增] 创新点新颖性验证
    ├── MCP 搜索学术文献，确认每个候选点是否已被提出
    ├── 输出: novelty_score (0-1)，过滤已被提出的点
    └── 降级: 搜索不到文献时，用 LLM 自评估（降低权重）
Phase 0.3: [新增] 创新点递进关系设计
    ├── 3个创新点必须有逻辑递进：
    │   点1: 问题定义/建模 → 点2: 方法创新 → 点3: 实验验证/应用
    ├── LLM 构建创新点之间的逻辑链
    └── 验证: 创新点链是否形成完整故事
Phase 0.4: [新增] 创新点与章节锚定
    ├── 每个创新点 → 明确映射到 Methodology 的哪个子节
    ├── 每个创新点 → 明确映射到 Experiments 的哪组实验
    └── 映射关系写入 bounded_context，强制执行
```

### 2.2 实验数据真实性

**现状**：实验数据从代码/报告中提取或由 LLM 生成。v8.0 输出中 Abstract 和 Experiments 存在数值不一致（MAE 0.125 vs 0.095）。

**问题根源**：
- `project_analyzer.extract_experiment_design()` 提取的数据不保证与实际代码运行结果一致
- LLM 在生成过程中可能"修正"数字，导致不一致
- 没有**单一数据源**（Single Source of Truth）

**改进方案**：

```
方案A: 从代码自动运行获取真实数据（如果代码可运行）
    ├── AblationDesigner 已能生成消融脚本，但当前是"可选"功能
    ├── 升级为必选: Phase 0.95 自动运行消融实验
    ├── 结果存入 unified_results.json（单一数据源）
    └── 所有章节从 unified_results.json 读取数据，禁止 LLM 编造数字

方案B: 无代码可运行时的降级
    ├── 从已有的训练日志/报告中提取数字
    ├── 用 MCP web-reader 读取 report.pdf 中的表格
    ├── 提取后存入 unified_results.json
    └── 无法提取时，明确标注 [DATA-PLACEHOLDER]，由用户填写
```

### 2.3 LaTeX 引用命令缺失

**现状**：v7.0 和 v8.0 输出中 LaTeX `\cite{}` 命令数量为 0。生成的 Markdown 中有 `[1]` 格式引用但没有转为 LaTeX 引用格式。

**根因**：CitationManager 的 `resolve()` 方法把 `<citation>` 标记解析为 `[N]` 格式（Markdown 风格），但没有转为 `\cite{key}` 格式。

**改进方案**：

```
CitationManager.resolve() 新增 LaTeX 模式:
    ├── 每个 verified citation 分配 \cite{authorYear} 格式的 key
    ├── 生成 .bib 文件条目
    └── 输出时根据 target_format (latex/markdown) 选择引用格式
```

## 三、从有到好：提升到顶刊水平（P1）

### 3.1 总结/评价模块

**当前架构**：
```
QualityGate (4维评估, 阈值70)
    → MultiReviewer (3个独立 Reviewer + Editor)
    → Auditor (反幻觉审计)
    → VERIFY (8项纯代码检查)
```

**差距**：

| 问题 | 描述 | 改进 |
|------|------|------|
| 评价维度不足 | 4维(学术规范/逻辑/引用/完整)太粗 | 扩展到 8-10 维，增加"创新性表达"、"故事线推进"、"与目标 venue 匹配度" |
| Reviewer 无领域知识 | 3个 Reviewer 是通用模板 | 注入 venue_profile 的审稿 checklist，让 Reviewer 按目标期刊标准评审 |
| 无对比基线 | 评价是绝对打分，不知道"好"的标准 | 引入 exemplar papers（同期刊已发表论文），让 Reviewer 对比评分 |
| 审稿意见不够具体 | 返回 issues 列表但缺少精确定位 | 要求 Reviewer 返回 (章节, 段落, 行号) 定位 |
| 缺少 meta-review | MultiReviewer 只综合不判断分歧 | 当 3 个 Reviewer 分歧 > 20分时，触发第4个 Reviewer 仲裁 |

### 3.2 生成模块

**当前架构**：
```
每个章节 → skill_orchestrator → LLM prompt → call_generation → 后处理
```

**差距**：

| 问题 | 描述 | 改进 |
|------|------|------|
| 章节生成是单次的 | 每章一次大 prompt，质量不可控 | 改为分段生成：每章先列要点 → 逐段生成 → 段间过渡 |
| 无段落级质量控制 | 质量门控是章节级 | 新增段落级检查：每段是否有明确目的、是否推进论点 |
| 缺少"写作声音" | LLM 生成的文字模板化 | 注入 writing_style_samples（从目标期刊的好论文中提取风格特征） |
| 数学公式无法验证 | 公式可能符号不一致或推导有误 | VERIFY 已有 V7 符号一致性检查，但需扩展为公式可证性检查 |
| 图表生成粗糙 | 消融图存在但信息密度低 | 学习 exemplar papers 的图表设计模式 |

### 3.3 控制模块

**当前架构**：
```
Dispatcher → plan_tasks() → 17个Phase → loop.run() → THINK→EXECUTE→VERIFY→REFLECT
```

**差距**：

| 问题 | 描述 | 改进 |
|------|------|------|
| Phase 是线性的 | 17个Phase 严格顺序执行 | 支持条件分支：如果创新点需要理论证明，插入"理论推导"Phase |
| 迭代效率低 | 每次重试都是整章重写 | 改为精准修复：只修改 Reviewer 指出的具体段落 |
| 无全局优化 | 章节各自独立迭代 | 新增 Phase 8（全局打磨）：通读全文，优化跨章节过渡、消除冗余、统一符号 |
| 降级策略简单 | API 失败→换模型→继续 | 降级时应该调整生成策略：弱模型用更短段落、更简单的句式 |
| 缺少进度反馈 | 用户只能看日志 | 实时展示当前阶段、质量分数、预计剩余时间 |

### 3.4 迭代模块

**当前架构**：
```
QualityGate: evaluate → revise → re-evaluate (最多3轮)
ChapterStateMachine: outline → draft → review → revision → final
```

**差距**：

| 问题 | 描述 | 改进 |
|------|------|------|
| 迭代终止条件单一 | score >= 70 就通过 | 增加多维条件：所有维度 >= 60 + 无 critical issues + 至少2维 >= 80 |
| 无自适应阈值 | 所有章节同一阈值 | Introduction/Conclusion 阈值可低，Methodology/Experiments 阈值应高 |
| 修改策略不智能 | revise prompt 是通用的"根据以下问题修改" | 分策略修改：结构调整 vs 内容补充 vs 语言润色 vs 数据修正 |
| 无学习机制 | 每次修改不积累经验 | 记录"修改→结果"对，下次遇到同类问题直接应用已知有效策略 |

## 五、执行-评价模型分离策略（已实现 v8.1）

### 设计原则
评价模型必须与执行模型来自**不同 provider**，避免"自我审查"盲区。

### 策略映射

| 可用模型 | 执行模型 | 评价模型 |
|---------|---------|---------|
| Qwen + GLM 都可用 | qwen3.7-max | glm-5.1（跨 provider）|
| 仅 Qwen 可用 | qwen3.7-max | qwen3.6-plus（同 provider 降级）|
| 仅 GLM 可用 | glm-5.1 | glm-5（同 provider 降级）|
| 仅 Token Plan | tp_qwen3.7-max | tp_deepseek-v4-pro（不同模型）|

### 涉及的评价模块
- `quality_gate.py` — `call_evaluation()` 替代 `call_reasoning()`
- `multi_reviewer.py` — Reviewer 和 Editor 均使用 `call_evaluation()`
- `auditor.py` — 反幻觉审计使用 `call_evaluation()`

## 七、架构优缺点总结

### 当前 v8.0 架构优点

1. **模块化程度高**：skill_orchestrator 模式使得每个章节独立，易扩展
2. **降级链健壮**：11级模型降级链 + 多源引用验证，不会因 API 故障卡死
3. **VERIFY 层零成本**：8项纯代码检查在 LLM 调用前拦截问题
4. **BoundedContext**：解决了上下文过长导致的质量下降问题
5. **执行-评价分离**：跨 provider 评价避免了自我审查盲区

### 当前 v8.0 架构缺点

1. **"先想后写"不够**：创新点确定是一次性的 LLM 调用，缺少多轮打磨
2. **"单次大生成"模式**：每章一个大 prompt → 一次生成，质量上限受限
3. **全局优化缺失**：各章独立迭代，没有通读全文的全局打磨
4. **无自学习能力**：每次运行从零开始，不积累经验
5. **过度工程化**：部分模块（如 SevenAnchorTest、RationaleMatrix）过于复杂但效果有限

## 八、结论

> **生成论文不难，难的是想清楚写什么。**

当前系统最大的差距不在于"写"，而在于"想"：
1. **想清楚创新点** — 需要"搜索-验证-递进"闭环，而非一次性 LLM 提取
2. **想清楚实验** — 需要真实数据作为单一数据源，而非 LLM 编造
3. **想清楚故事** — 需要全文级的叙事规划，而非各章独立生成

v9.0 的核心目标应该是**"想清楚"**——让系统在写之前就把创新点、实验、故事线全部想清楚，然后用高质量的多段生成+精准迭代把"想清楚的"写出来。
